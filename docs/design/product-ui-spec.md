# 产品 UI 设计规范

## 1. 目的

这份规范提炼自当前 Pony Agent 已经落地的界面语言，用来指导整个产品后续页面的 UI/UX 设计与改造。

适用范围：

- 首页工作台
- Provider / Model 配置页
- Runtime、Session、Trace、Tool、Diagnostics 类页面
- 后续新增的产品页面与前端模块

目标不是做“视觉效果相似”，而是让整个产品在结构、节奏、密度、交互和信息表达上属于同一套系统。

## 2. 总体气质

关键词：

- 极简
- 克制
- 温暖
- 工程化
- 可长时间使用

界面应避免“产品宣传页感”或“高饱和控制台感”，要像一个稳定、安静、可持续使用的工程工作台。

## 3. 视觉方向

### 3.1 色彩

主色系为暖米白、浅沙色、石色、棕灰色。

推荐角色分配：

- 页面背景：暖米白渐变、很轻的暖色氛围，不用纯白平铺
- 主面板：半透明或轻微染色的白色/米白色卡片
- 强调底色：`#f6efe3`、`#f8f4ee` 这一类暖米色
- 文字主色：深石色、深棕灰，而不是纯黑
- 分割线：低对比浅石色
- 状态色：仅在错误、警告、成功等语义位置出现，不拿来做主题色

避免：

- 冷白 + 冷灰主导全局
- 紫色、蓝紫色作为默认品牌强调色
- 高饱和大面积品牌色块
- 纯黑大面积按钮和纯白页面背景的强烈对撞

### 3.2 明度关系

层次优先通过明度和留白建立，而不是靠厚重阴影和强边框。

推荐顺序：

- 页面背景最轻
- 面板略深
- 局部强调块再略深
- 文字和图标最深

## 4. 布局规范

### 4.1 页面骨架

当前产品默认采用工作台式骨架，主页为三栏工作台：

- 左：会话 / Session 导航
- 中：主对话与输入区
- 右：状态、Trace、Diagnostics

后续页面可以在这个基础上演化成双栏配置页或单栏详情页，但都要保持同一套结构纪律。

规则：

- 页面容器固定在视口内，不允许整体内容无限向下撑高
- 主内容区必须是 `h-full` + `min-h-0` 链路
- 需要滚动的内容，应在内部面板滚动，而不是整页滚动
- 视口底部保留轻量留白，当前规范为 `pb-2`

### 4.2 面板关系

同一屏中的主要面板底部应对齐，并共同撑满到当前视口底部。

规则：

- 左栏、中栏、右栏都是“固定高度容器 + 内部滚动”
- 面板外层使用统一圆角
- 面板间通过 `gap` 留白分隔，不靠重边框分隔

### 4.3 圆角

默认使用中小圆角，不要极大圆角。

建议区间：

- 主面板：`0.6rem` 左右
- 子卡片：`0.45rem`
- 徽标、轻按钮：`0.35rem` 到 `9999px`

避免：

- 过大的胶囊卡片
- 同一页混用太多圆角体系

## 5. 间距与密度

### 5.1 默认密度

这不是宽松营销页，而是高信息密度产品工作台。

规则：

- 面板内留白克制但不拥挤
- 关键结构之间保留明确节奏
- 细节层级允许更紧凑，尤其是 Trace、Tool、Step、Metadata

### 5.2 信息密度层级

建议按三层控制密度：

- 一级区域：标题、栏切换、主面板，留白较宽
- 二级区域：卡片主体、表单块、消息块，中等密度
- 三级区域：Trace Step、Tool 明细、Token/时长标签，紧凑密度

避免：

- 所有层都同样疏松
- 所有层都同样拥挤

## 6. 字体与文字层级

### 6.1 字体气质

优先使用简洁、稳定、工程感强的无衬线体系。

文字风格应具备：

- 标题清晰
- 辅助信息弱化
- 英文、ID、模型名、协议名在混排时仍然整洁

### 6.2 层级规则

默认分层：

- 页面主标题：高权重，大字号，紧字距
- 面板标题：中高权重，小于页面主标题
- 区块标签：小号、uppercase、增加 tracking
- 正文：常规字号，强调可读性
- 元信息：更小字号，低对比

避免：

- 全页所有文字一个字号体系
- 说明文字和主标题抢视觉权重

## 7. 组件规范

### 7.1 面板

面板是当前产品 UI 的核心基本单位。

规则：

- 主面板使用轻背景 + 微染色
- 面板内部块使用暖米色浅底区分
- 不依赖大阴影表现层级

### 7.2 按钮

按钮分三类：

- 主动作：克制但明确，数量少
- 次动作：浅底、低对比
- 轻动作：ghost 风格，主要用于切换、删除、编辑等局部操作

规则：

- 同一区域只保留一个真正主动作
- 删除操作通常用 ghost 或轻按钮，不要做成最强视觉按钮

### 7.3 徽标与标签

Provider/Model 徽标、Token 标签、协议标签都应使用页面同色系。

规则：

- 优先暖米色底 + 棕灰文字
- 小尺寸、低侵入
- 标签是说明，不是视觉主角

### 7.4 表单

表单应看起来像工作台配置，而不是营销式设置页。

规则：

- 按逻辑分组
- 每组有标题和必要说明
- 输入框和下拉框风格一致
- 背景通常为白色或浅米色，不做重描边

当前 Provider 配置页的协议选择应放在表单内部，而不是放在新增入口做路径分叉。

## 8. 消息区规范

### 8.1 对话区

主对话区是产品中的核心舞台之一。

规则：

- 用户消息和 Agent 消息视觉上明显区分
- Agent 内容主体优先文字可读性
- 第一行元信息简洁，不喧宾夺主
- 输入区固定在底部，历史消息区内部滚动

### 8.2 Markdown

规则：

- 行高适度压紧，避免松散
- 代码块与正文风格统一
- 列表、引用、代码块遵循同一暖米色体系

## 9. Trace / Diagnostics 规范

### 9.1 设计目标

右栏不是装饰栏，而是工程诊断面板。

规则：

- 优先展示状态、原因、执行轨迹
- 信息要紧凑，但不能乱
- 折叠层级要清楚

### 9.2 手风琴规则

当前产品已经建立的规范：

- 外层面板可折叠
- Trace 内部使用手风琴
- 同层只展开一个主要折叠项

这样做是为了控制高度和信息噪声。

### 9.3 密度规则

Trace 的二三级内容可以比普通卡片更紧凑。

适用对象：

- Step 列表
- Tool 卡片
- Arguments / Result 明细
- Token、时长、协议等元信息

## 10. 滚动规范

整个产品优先内部滚动，不优先整页滚动。

规则：

- 左栏、中栏、右栏内容超高时，都应在各自容器内部滚动
- 滚动条统一使用 `shadcn-vue` / `reka-ui` 的 `ScrollArea`
- 纵向滚动条可见，横向滚动默认禁止
- 如果内容可能超宽，优先强制换行，不优先横向滚动

例外：

- 代码块可以按需局部横向滚动
- 但 Trace、表单、状态文本、说明文字不应引入横向滚动

## 11. 动效规范

动效应该轻，不应该“秀”。

规则：

- 折叠、展开、内容流入使用短时长过渡
- 以淡入、位移、轻量高度过渡为主
- 避免明显弹性、夸张缩放、重动效

推荐感觉：

- 平滑
- 安静
- 服务于理解

## 12. 页面改造时的执行顺序

后续页面改造按下面顺序做，而不是先堆局部样式：

1. 先定页面骨架：是否属于三栏工作台、双栏配置页、单栏详情页
2. 再定滚动模型：整页滚还是面板内滚
3. 再定信息层级：标题、说明、正文、元信息
4. 再定组件语气：按钮、标签、卡片、表单
5. 最后补局部细节：颜色、间距、动效

## 13. 不应做的事

- 不要给每个区域都上强阴影
- 不要引入新的高饱和主色
- 不要把配置页做成“产品官网设置页”
- 不要让整页无限向下增长
- 不要在 Trace / Diagnostics 中默认横向滚动
- 不要让标签、徽标、状态小块抢过正文和结构
- 不要在没有理由时脱离当前产品已建立的米白暖石色体系

## 14. 页面评审清单

后续任何前端 UI/UX 改动，至少自检以下问题：

- 它是否仍然属于当前产品同一套暖米色极简体系
- 它是否优先采用内部滚动，而不是把整页继续撑高
- 它的面板圆角、留白、边界处理是否与现有产品一致
- 它的标题、标签、元信息层级是否清楚
- 它的状态、Trace、Tool、诊断信息是否紧凑但不混乱
- 它有没有不必要的新颜色、新按钮语气或新视觉语法
- 它的交互是否安静、明确、无多余花哨效果

## 15. 约束说明

从现在开始，Pony Agent 仓库中涉及前端 UI/UX 的改动，默认应遵循本规范。

如果需要偏离，应该：

- 明确指出偏离点
- 说明为什么当前产品规范不适用
- 说明新页面采用的新局部规则
