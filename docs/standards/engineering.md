# 工程规范与最佳实践

## 总原则

- 小步前进
- 先清晰后复杂
- 先可运行后可扩展
- 先可观察后可优化

## 目录规范

- `hermes/` 只作参考
- 根目录工程是 Pony Agent 主实现
- 架构、规范和路线图写入 `docs/`

## 编码规范

- 优先可读性
- 避免过早抽象
- 一个模块只负责一件事
- 重要设计选择要写明原因

## 重构规范

- 不以“尽快复刻 Hermes 全量能力”为目标
- 先做最小主链
- 新增抽象时必须回答：
  - 它解决了什么问题
  - 它的边界是什么
  - 如果没有它会怎样

## 测试与验证

- 每完成一个阶段，至少保证一条主流程可运行
- UI 要能展示核心运行状态
- 不把“看不见的正确”当成完成

### 五层质量门

Pony Agent 的“校验层”不等于只跑测试。默认按以下 5 个 gate 组织实现前约束、实现中校验和完成态证据：

1. `Gate 1 / 生成前约束`
   - 先明确目标、范围、非目标、接口边界、不可修改区域和完成定义。
   - 需求不清、验收口径不明、边界未冻结时，不应直接进入实现。
2. `Gate 2 / 静态质量门`
   - 覆盖格式、lint、类型检查、编译、schema/contract、依赖与基础安全扫描。
   - 目标是先排除“根本不可合并”的低成本错误，再进入行为验证。
3. `Gate 3 / 行为质量门`
   - 覆盖单测、集成测试、回归测试、端到端测试、契约测试或最小可运行主链验证。
   - 目标是证明变更行为正确，而不是只证明代码能编译。
4. `Gate 4 / 系统质量门`
   - 覆盖性能、资源占用、并发正确性、可观测性、安全性、兼容性、稳定性等非功能约束。
   - 对桌面端能力、跨端行为、持久化与恢复链路，默认要关注这一层。
5. `Gate 5 / 发布质量门`
   - 覆盖变更说明、迁移要求、feature flag、回滚路径、监控点、已知风险与上线约束。
   - 目标是确保“可发布、可回滚、可排障”，而不是只在本地通过。

### 执行顺序

- 默认顺序是：先过 `Gate 1`，再跑 `Gate 2`，再跑 `Gate 3`，最后视任务风险补 `Gate 4` 和 `Gate 5`。
- 校验失败时，先定位失败属于哪一层，再做最小修复，不要用扩大改动范围的方式“碰运气过关”。
- 任何一次失败，如果证明当前仓库缺少长期 guardrail，修复后应补对应测试、规则或文档约束，避免同类问题再次出现。

### 完成态要求

- 任务卡、session log、review 文档中的“验证”或“验收证据”，默认应标明本次变更实际覆盖了哪些 gate。
- 轻量任务可以不覆盖全部 5 层，但必须显式说明本次跳过了哪些 gate，以及为什么当前风险可接受。
