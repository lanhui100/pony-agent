# 0007 将缓存命中提升为一等产品指标，但按阶段落地

## 背景

Pony Agent 正在从单轮 runtime、工具调用、多轮会话逐步演进到 graph 编排与高级能力。

在这条路线里，缓存命中不是一个局部优化项，而是工业级 agent 产品的核心成本控制点：

- 它直接影响单位任务成本
- 它直接影响首 token 延迟和整体吞吐
- 它直接影响项目是否具备可规模化的商业价值

尤其在 DeepSeek 这类强依赖前缀复用收益的模型上，缓存命中不应被视为“后期再看”的性能细节。

但与此同时，缓存命中优化也不能脱离当前实现阶段，提前为尚不存在的能力做复杂设计，导致工程节奏失衡。

## 决策

我们做出以下决策：

### 1. 缓存命中提升为一等优先级

在不牺牲 agent 输出质量的前提下：

- 缓存命中是运行时架构设计的第一优先级之一
- 所有上下文管理、摘要、压缩、历史裁剪、子代理、graph 会话设计，都必须显式评估对缓存命中的影响
- 任何会长期破坏稳定前缀的方案，都不能只因为“功能可用”而被默认接受

这条原则适用于：

- provider 请求构造
- 会话历史结构
- compaction / summary
- tool follow-up
- planner / executor 协作
- 子代理 / fork / graph run 会话边界

### 2. 缓存命中必须按阶段落地，而不是抢跑

缓存命中是全局约束，不是所有阶段都要立刻实现完整优化。

我们的原则是：

- 在早期阶段，优先避免做出未来难以修正的“反缓存”结构
- 在相关能力尚未形成前，不提前做复杂缓存子系统
- 一旦某类能力进入主线，就必须同步定义其缓存友好约束

也就是说：

- 早期关注“不要埋雷”
- 中期关注“建立边界和观测”
- 后期关注“系统化优化与调度”

### 3. 输出质量优先于缓存命中，但缓存命中优先于便利性实现

排序明确为：

1. 输出质量与任务正确性
2. 缓存命中与单位成本
3. 实现便利性

因此：

- 不能为了命中率牺牲主任务连续性和回答质量
- 也不能因为实现简单，就接受持续破坏缓存前缀的架构

## 分阶段执行原则

### Phase 2：单轮 Runtime

目标不是优化命中率，而是避免形成错误结构。

要求：

- system prompt、provider capability、工具定义的组织方式尽量稳定
- 不在请求前部注入频繁变化的临时状态
- trace 中开始记录基础 token usage 与 cached token usage

此阶段不做：

- 自动 compaction
- 复杂摘要重写
- planner / executor 双会话优化

### Phase 3：工具调用

重点是避免 tool follow-up 污染主前缀。

要求：

- 工具结果注入方式可压缩、可替换、可观测
- 明确区分稳定前缀与 follow-up 动态尾部
- 开始按“首请求 / follow-up”拆分缓存命中指标

此阶段不做：

- 全量历史重写
- 高频自动摘要

### Phase 4：会话与记忆

这是缓存命中正式进入主线设计的第一阶段。

要求：

- 建立稳定层、半稳定层、易变层
- 会话摘要默认冻结，禁止每轮重写
- 首版 compaction 仅允许在明确边界发生
- 历史裁剪必须避免每轮重新切边界

这是第一阶段必须真正交付缓存友好结构，而不只是观测。

### Phase 5：Graph 编排

这是第二阶段必须系统优化缓存命中的阶段。

要求：

- graph run、planner、executor 的会话边界显式化
- 不同角色尽量拆分为独立会话或独立稳定前缀
- run state / retrieved context / checkpoint 信息不能污染主前缀

### Phase 6 以后：高级能力

要求：

- 子代理默认走独立上下文或可复用 fork 前缀
- 高级 workflow / 并行 agent / 背景任务都必须带缓存成本设计
- 引入 cache guard、回归测试和阶段性预算门槛

## 结果

这个决策意味着：

- 缓存命中不再是“性能优化 backlog”
- 它成为 Pony Agent 主线架构的设计约束
- 但它的具体实现必须和项目阶段同步推进

一句话概括：

缓存命中是商业价值的一号成本指标，但工程实施必须按能力成熟度逐阶段推进。
