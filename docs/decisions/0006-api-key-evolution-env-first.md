# 0003 API Key 存储先采用环境变量，后续再演进到安全凭证层

## 背景

Pony Agent 当前处于学习式重构阶段，主线目标是持续打通这些核心能力：

- `run_turn()` 的真实执行链路
- 多 provider / 多模型切换
- tool routing 与 tool execution
- Rust runtime 与桌面工作台之间的协作

与此同时，项目已经开始进入真实 provider 接入阶段，因此 `API Key` 的来源与存储方式必须尽快有一个明确结论。

## 约束

- `API Key` 是敏感信息，不适合继续作为普通前端状态长期保存
- 当前阶段不能让密钥系统设计阻塞 agent core 的主线推进
- Pony Agent 未来需要考虑多端部署，因此最终方案不能只绑定桌面端本地存储
- 当前团队仍处于学习和快速验证阶段，优先级是“先跑通主链路，再逐步加固”

## 决策

当前阶段，`API Key` 的默认来源先采用环境变量（`env`）。

具体策略是：

- 非敏感配置继续保存在 Rust 管理的配置文件中
- `API Key` 不作为长期明文配置写入普通 JSON
- Rust provider 解析时，当前阶段优先使用运行时传入值，其次读取环境变量
- 安全凭证层作为下一阶段演进目标，而不是当前阻塞项

当前推荐优先级为：

`runtime input > env`

后续如果引入独立凭证层，再演进为：

`runtime input > secure credential store > env`

## 为什么这么做

这样做的核心原因不是“环境变量最安全”，而是“环境变量在当前阶段最直接、最便宜、最不阻塞主线”。

它的价值在于：

- 实现简单，Rust 侧天然易于读取
- 不会打断当前对 provider、runtime、tool call 的连续验证
- 对学习阶段更友好，便于先理解 provider 配置与运行时注入的关系
- 后续可以平滑升级，不会推翻当前 provider 抽象

## 权衡

### 好处

- 最快落地
- 与当前 Rust provider 结构天然兼容
- 便于本地开发和多 provider 调试
- 不会让安全方案反过来阻塞 agent 核心演进

### 代价

- 环境变量不是最终的安全存储方案
- 对普通桌面用户不如系统凭证存储直观
- 如果未来扩展到 Web 或托管部署，还需要进一步分层治理

## 后续影响

这项决策意味着 Pony Agent 接下来需要坚持以下边界：

- agent core 不直接承担密钥存储职责
- 普通配置与敏感凭证继续分层
- 未来多端部署时，以统一凭证接口适配不同平台实现，而不是强求单一底层存储方式

## 后续演进方向

后续可按以下顺序演进：

1. 当前阶段：`env` 作为最小可用方案
2. 桌面端阶段：引入系统安全凭证存储
3. 多端阶段：抽象统一的 `CredentialStore`
4. 服务端 / Web 阶段：分别接入 secret manager 或服务端代理
