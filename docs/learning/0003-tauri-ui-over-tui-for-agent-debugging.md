# 0003 Rust 智能体核心调试为什么优先用 Tauri UI 而不是 TUI

## 问题

做 Rust 重构智能体核心时，用 TUI 还是 Tauri UI 来做效果测试和交互？

## 简短结论

当前阶段优先用 Tauri UI，因为我们更需要“看清运行时状态”和“可视化调试”，而不是先复刻终端交互体验。

## 系统化梳理

### 1. TUI 的价值

TUI 适合：

- 远程终端
- 轻量操作
- SSH 场景
- 开发者工具入口

但它会带来很多额外工作：

- 输入输出交互
- 滚动和重绘
- 快捷键
- terminal 兼容性

### 2. Tauri UI 的价值

Tauri UI 更适合当前阶段，因为它能直观展示：

- 当前运行阶段
- Graph 节点切换
- 工具调用参数和结果
- 健康状态
- Turn 轨迹

也就是说，它更像一个“智能体运行时调试台”。

### 3. 为什么这对学习模式更重要

学习式重构最怕的不是做不出来，而是“只看到结果，看不到过程”。  
Tauri UI 更容易把过程暴露出来，让我们理解系统是怎么工作的。

## 常见误区

- 误区 1：智能体项目就应该先做 TUI，因为看起来更像 CLI
- 误区 2：UI 只是产品包装，调试靠日志就够了
