# 对话历史侧栏与会话管理 UI 的设计取舍

## 问题

为什么我们现在要把“对话历史”放到左侧，并做成可折叠、可新建、可清除的侧栏，而不是继续把它塞在右侧状态区里？

## 简短结论

因为 `session` 已经不只是后端数据结构，而是用户可管理的工作流对象了。

把它单独放在左侧，能更清楚地表达：

- 中间是当前对话
- 左边是历史与入口
- 右边是状态和 trace

这比把所有信息混在一个侧栏里，更符合我们现在的重构阶段。

## 系统化梳理

### 1. 为什么要单独做“对话历史”

因为 session 已经有了明确边界：

- Rust core 持有真实会话状态
- 前端只是切换和展示 session
- 每个 session 都是独立的上下文容器

所以 UI 也应该把它显式画出来，而不是隐藏成一个普通列表项。

### 2. 为什么放左侧

左侧更适合放“入口型”和“导航型”内容。

右侧更适合放“观察型”和“诊断型”内容。

这就是我们现在的分工：

- 左侧：对话历史
- 中间：对话区
- 右侧：状态 + Trace

### 3. 为什么要可折叠

因为 session 列表会长，而且工作时未必一直需要看。

可折叠的好处是：

- 需要时能快速切换
- 不需要时能把空间让给对话区
- 更接近一个 workbench，而不是固定三栏面板

### 4. 为什么允许清除

因为历史对话不是装饰，它是可管理的状态。

如果不能清除，就意味着：

- 测试会话会越堆越多
- 浏览器预览和桌面联调的状态会越来越脏
- 用户无法维护自己的工作流

所以“清除”是 session 管理的一部分，不是附加功能。

### 5. 为什么状态栏要更克制

状态栏的职责是读当前状态，不是当导航区。

因此它更适合展示：

- 当前 session id
- phase
- provider / model
- token / latency
- trace

而不是承载会话列表。

## 常见误区

- 误区 1：会话列表只是一个侧栏装饰
- 误区 2：折叠只是为了省空间
- 误区 3：清除会话只是删除 UI 数据
