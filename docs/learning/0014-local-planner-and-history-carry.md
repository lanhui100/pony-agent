# 0014 本地 planner、跨轮历史与工作区工具为什么要前置

## 这次学习回答了什么

这一轮不是在补一个孤立 bug，而是在回答三个更底层的问题：

1. 为什么有些工作区请求不应该先打远端 decision
2. 为什么“这个文件第 20 行”这类问题会失败
3. 最小可用的多轮上下文，应该先落在哪一层

## 先说结论

这一轮最大的收获是：对于明显的工作区类请求，Pony 不能把“是否要用工具”完全外包给 provider。

至少在当前阶段，更稳的做法是：

- 先在本地做一层轻量 planner
- 能直接判断成 `workspace.*` 的请求，就直接进工具
- 再把工具结果交给真实 provider 做 follow-up 生成

这样做并不是否定原生 tools 协议，而是在当前产品阶段，把“工具判定稳定性”优先放在 Pony 自己手里。

## 这次暴露出的两个真实问题

### 1. 明显的工作区请求先白等 45 秒

之前像“当前文件夹下有哪些文件？”这种问题，本质上根本不需要远端模型先做复杂决策。

但旧链路还是会先发一轮 decision 请求，于是出现了：

- provider decision 45 秒超时
- fallback 到本地 mock
- 然后才由本地工具兜底继续执行

这说明当前失败点并不在工具，而在 decision 阶段的职责划分过重。

### 2. “这个文件第 20 行”这类跨轮指代会丢失

第二个问题更关键：

- 当用户先问 `tauri.conf.json` 是什么
- 再问“它的第 20 行是什么”

如果后端没有会话历史，它只能把这条消息当成一个孤立句子，自然不知道“它”指的是谁。

这不是工具坏了，而是最小多轮上下文还没有接到 runtime。

## 为什么要引入本地 planner

这里要特别分清：本地 planner 不是在重新发明一套 provider 协议，而是在做“高确定性请求的前置分流”。

这一层最适合处理的，是这些明显模式：

- 列目录
- 读具体文件
- 读某个文件的某几行
- 根据上一轮已经出现的文件名继续追问

这些请求的共同点是：

- 意图非常窄
- 命中工具的确定性高
- 如果先走远端 decision，收益很小，延迟和失败面却更大

所以本地 planner 的意义，不是替代模型，而是缩短工具型请求到工具执行之间的路径。

## 为什么跨轮历史要先接到 runtime，而不是先做数据库

这一轮也进一步说明了一件事：

“多轮上下文”与“持久化存储”不是一回事。

当前先解决的是：

- 当前会话里，下一轮能否理解上一轮提到的文件

这属于 runtime 输入问题。

它只需要：

- 前端把最近几轮消息带给后端
- 后端在 planning 与本地工具推断时使用这段历史

它还不要求：

- SQLite
- Postgres
- Redis
- 跨重启恢复

也就是说，最小多轮上下文应该先在 runtime 层打通，再决定是否落持久化。

## 这次修复实际做了什么

这一轮落地了三件事：

### 1. 本地 planner 前置

对明显的工作区请求，先做本地工具判定：

- `workspace.list_files`
- `workspace.read_file`
- `workspace.read_file_segment`

这样像“当前文件夹有哪些文件”“这个文件是什么”“第 20 行是什么”不再需要先走远端 decision。

### 2. 前端把最近几轮 history 带到后端

当前 `submitTurn()` 已经会把最近几轮 user / assistant 消息一起传给 Rust runtime。

这意味着后端在 planning 阶段终于不再是“每轮失忆”。

### 3. 本地工具推断支持文件回溯和行号读取

后端现在已经能处理两类典型追问：

- 从最近用户消息里回溯上一个文件名
- 从“第 N 行”推成 `read_file_segment`

比如：

- `tauri.conf.json 这个文件是什么文件？`
- `它的第 20 行是什么？`

这条链现在已经能在本地稳定命中。

## 这次还踩到了什么坑

最有代表性的坑是“历史回溯过宽”。

一开始回溯上一个“文件名”时，规则过宽，结果把 assistant 回复里的版本号 `0.1.0` 误当成了文件名，于是出现：

- `workspace.read_file_segment(path="0.1.0")`

这个问题非常有启发性，因为它说明：

- 多轮历史不是“带上就行”
- 还要区分哪些历史适合参与工具推断

所以这一轮最后又把规则收紧成：

- 文件回溯只看最近用户消息
- 不再从 assistant 回复里抽文件名

## 这对架构意味着什么

这一轮之后，Pony 的链路比之前更像一个真正的 agent runtime：

- provider 负责真实生成与原生 tools 协议
- runtime 负责回合推进
- 本地 planner 负责高确定性工具前置判定
- history 负责最小多轮语境
- tool router 负责最终执行

这里已经能看到未来更完整 query loop 的雏形了。

也就是说，Pony 不再只是“把一条消息送给模型”，而是开始形成：

- 本地判断
- 工具执行
- 模型跟进
- 多轮延续

这一整条运行链。

## 当前还没有的部分

虽然这轮已经把最小多轮工具链打通，但还没到完整会话系统：

1. history 仍然主要来自前端内存态
2. 还没有真正的 `SessionStore`
3. 还没有把“上一轮引用的文件”沉淀成结构化会话状态
4. 还没有跨重启持久化
5. 还没有更复杂的多工具跨轮规划

所以当前更准确的说法是：

Pony 已经有“最小跨轮工具语境”，但还没有“完整会话记忆系统”。

## 当前学习进度

截至这一轮，可以明确地说：

1. Pony 已经有原生 tools 主路径
2. Pony 已经有本地 planner 前置能力
3. Pony 已经能把最近几轮历史带入 runtime
4. `workspace.list_files / read_file / read_file_segment` 已经形成可用的最小工作区工具组
5. “文件 -> 继续问该文件的某一行”这条真实工作流已经打通

## 下一步最自然的方向

顺着这一轮往下走，最自然的下一步是：

1. 把“当前引用文件”从临时 history 推断，升级成更稳定的会话状态
2. 继续补更多工作区工具，比如搜索、按模式查找、目录递归读取
3. 把 token 统计、首 token 延迟等运行指标补到 UI
4. 设计 `SessionStore`，区分“当前前端内存态 history”和“真正的持久化会话”

## 一句总结

这一轮的本质，不是修了几个工具 bug，而是让 Pony 从“单轮能调工具”推进到“在最小多轮语境里，稳定地判断并执行工作区工具”。
