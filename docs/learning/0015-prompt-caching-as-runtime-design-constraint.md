# 0015 为什么要把缓存命中纳入 runtime 重构约束

## 这次学习回答了什么

这一轮不是在讨论某一家 provider 的计费技巧，而是在回答一个更基础的问题：

- 为什么 prompt caching 不应该只被当成“成本优化”
- 为什么它会反过来影响 runtime、history、tools 和 session 的设计
- Pony Agent 应该把缓存命中放进哪一层的设计约束里

## 先说结论

缓存命中不只是 provider 的能力，也是 agent runtime 的结构性约束。

原因很简单：

- provider 是否命中缓存，强依赖请求前缀是否稳定
- 而请求前缀是否稳定，取决于我们如何组织 system prompt、developer prompt、工具清单、history、tool result 和会话摘要

也就是说，缓存命中看起来发生在 provider 端，但决定它好坏的，往往是我们自己在 runtime 层的设计方式。

## 为什么这件事必须纳入重构考量

如果不把缓存命中提前当成架构约束，后面很容易出现这些问题：

1. 每轮都重写前缀，导致 provider 几乎无法命中缓存
2. history 压缩方式不稳定，今天一种、明天一种，前缀持续抖动
3. 工具清单顺序、格式、注释频繁变化，缓存不断失效
4. session state 虽然越来越完整，但请求代价和延迟越来越差

这样最终会导致一种典型反效果：

- 系统逻辑越来越“聪明”
- provider 侧却越来越“贵”和“慢”

所以缓存命中不是后期优化项，而是应该在 runtime 重构阶段就一起考虑。

## 放到 Pony Agent 当前架构里，应该怎么理解

当前 Pony 已经形成了这样的链路：

- system / developer 指令
- 工具清单
- 最近几轮 history
- 当前用户问题
- 必要时的 tool result follow-up

从缓存视角看，这几层并不等价。

### 应该尽量稳定的部分

- system prompt
- developer 规则
- 工具清单本身
- 工具清单顺序与格式
- 较长期稳定的会话摘要

这些更像“请求前缀骨架”，越稳定，越容易命中 provider cache。

### 应该允许变化的部分

- 当前用户消息
- 当前工具结果
- 本轮临时说明

这些天然属于后缀，变化本来就正常，不应该强求缓存。

### 当前最敏感的中间带

- 最近几轮 history
- history 压缩策略
- “上一轮文件引用”这类上下文补丁

这里最容易失控，因为它既不是完全静态，也不能无限增长。

也正因为如此，history/session 设计会直接影响缓存命中率。

## 当前阶段对 Pony 的具体启发

这一轮之后，可以明确把缓存命中纳入这些设计原则：

### 1. 能本地前置的工作区请求，尽量不要先打远端 decision

这不只是为了减少超时，也是在减少不必要的 provider 请求。

本地 planner 前置越合理，越少会把“明明本地就能确定”的请求送去消耗一次远端 prefill。

### 2. history 不应每轮大改

最坏的做法是：

- 每轮都把历史重写成另一种摘要
- 每轮都插入格式不同的 developer 注释
- 每轮都重排工具说明

这会让请求前缀不断抖动，provider 很难命中缓存。

### 3. 工具清单应该稳定注入，而不是混进历史反复累积

工具清单应当是：

- 每轮请求里的当前能力描述
- 顺序稳定
- 结构稳定

但不应该是：

- 被当成普通历史消息不断回写

否则既浪费上下文，也破坏缓存。

### 4. session store 未来不只是存消息，还要服务“稳定前缀”

未来设计 `SessionStore` 时，不能只想“把历史存下来”。

还要想：

- 哪些内容可以固化成较稳定的 session summary
- 哪些内容应该留在实时增量区
- 哪些内容不能频繁重写，否则会打碎缓存前缀

## 这意味着下一阶段不能只看功能正确

后面做重构时，至少要同时看三条轴：

1. 功能是否正确
2. 运行时语义是否清楚
3. 请求前缀是否足够稳定，利于 provider cache

也就是说，未来一个设计方案即使“能跑”，如果它让：

- history 结构频繁变化
- 工具定义不稳定
- provider 请求前缀每轮都重建

那它在 Pony 里也不算好方案。

## 当前还没有做的部分

虽然已经把缓存命中纳入考量，但当前还没真正落到工程指标上：

1. 还没有显示 provider 侧的 cache hit / miss 指标
2. 还没有为 session summary 设计“稳定前缀优先”的策略
3. 还没有为工具清单做严格的顺序和结构冻结约束
4. 还没有区分“利于缓存的长期上下文”和“只服务当前回合的短期上下文”

所以当前更准确地说，是：

Pony 已经把缓存命中纳入设计维度，但还没有把它产品化成可观测指标和硬约束。

## 下一步最自然的动作

顺着这条线，后面最自然的动作是：

1. 在任务系统中把“稳定前缀 / cache-friendly history”明确写成约束
2. 设计 `SessionStore` 时明确区分长期上下文和短期上下文
3. 约束工具清单格式、顺序和注入点，减少前缀抖动
4. 如果 provider 暴露缓存统计，后续把 hit/miss 纳入运行面板

## 一句总结

缓存命中不是 provider 的附加优惠，而是会反向塑造 Pony runtime、history 和 session 设计方式的一条硬约束。
