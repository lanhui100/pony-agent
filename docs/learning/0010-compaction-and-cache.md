# 0003 上下文压缩与 Cache 命中

## 这次要回答的问题

这次学习关注的是一个非常工程化的问题：

- 在 agent 架构里做上下文压缩，会不会影响 KV cache 或 prompt cache 命中
- 如果会，为什么这会直接影响回复速度和成本
- 以后我们设计 Pony Agent 的压缩策略时，应该优先注意什么

## 先说结论

结论是：会，而且影响很大。

需要记住的核心判断是：

- 任何会改变前缀 token 序列的压缩，都可能降低 cache 命中
- 压缩虽然能减少上下文长度，但不一定更便宜
- 如果压缩导致稳定前缀反复变化，反而可能更慢、更贵

## 为什么压缩会影响 cache

无论是 KV cache 还是更高层的 prompt cache，本质上都依赖一件事：

前缀 token 序列尽量稳定。

如果前缀保持不变，模型就能复用已经计算过的部分，从而减少重复计算。

但压缩通常会做这些事：

- 删除旧消息
- 把多轮历史改写成摘要
- 合并或重排消息结构
- 改写 tool result 的表示

从语义上说，这些操作是在“保留信息”。

但从 cache 的角度说，这些操作常常是在“重写前缀”。

cache 关心的是 token 是否一致，而不是语义是否大致相同。

## 为什么这会影响速度和成本

一旦前缀变化太大，cache 命中率就会下降。

这会带来几个后果：

- 首 token 更慢
- 每轮重复计算更多
- 高价输入 token 变多
- 低价缓存命中 token 变少

所以不能简单理解成：

“上下文更短 = 一定更省”

真实情况经常是：

- 上下文变短了
- 但前缀稳定性变差了
- 每轮反而需要重算更多内容
- 综合下来不一定更便宜

## 一个直观例子

假设原始历史是连续增长的：

- system
- user A
- assistant A
- user B
- assistant B
- user C

如果每一轮只是继续往后追加一点内容，那么前面的前缀大体稳定，cache 更容易命中。

但如果压缩后变成：

- system
- summary of A + B
- user C

虽然字面上更短了，但前缀已经被改写成另一套 token 序列。

这时原来那段可复用的缓存前缀，很可能就失效了。

## 这对 Claude 类系统意味着什么

这也是为什么像 Claude Code 这种 agent runtime，不会粗暴地“长了就压一遍”。

从代码里能看到它用了很多更细的机制，比如：

- microcompact
- autocompact
- context collapse
- compact boundary
- tool result budget

这说明它真正解决的问题不是“怎么压短”，而是：

- 哪些内容必须保留稳定
- 哪些内容可以压缩
- 什么时机压缩才值得
- 压缩后怎样尽量不破坏后续 cache

## 对 Pony Agent 的启发

以后我们为 Pony Agent 设计压缩策略时，不能只看“窗口快满了”，还要看：

- 这次压缩会不会打断稳定前缀
- 压缩结果是不是会在下一轮又被重写
- 有没有可能把旧摘要固定下来，避免反复改写
- 能不能只压某一层，而不是重写整个历史

## 一个实用判断问题

后面每次设计压缩策略时，都可以先问：

“这次压缩虽然缩短了上下文，但它是不是破坏了本来可以长期复用的稳定前缀？”

如果答案是“是”，就要非常小心。

## 当前学习结论

对于 agent runtime 来说，好的压缩不是单纯追求“更短”，而是追求：

- 信息仍然够用
- 前缀尽量稳定
- cache 尽量命中
- 成本和延迟综合更优

这意味着“压缩策略”不是纯粹的上下文管理问题，它同时也是性能设计和成本设计问题。
