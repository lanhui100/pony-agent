# 0002 为什么前端选 Vue 3 + Pinia，暂不上 vue-router

## 问题

Pony Agent 前端为什么选 `Vue 3 + TypeScript + Vite`，并且建议上 `Pinia`，但第一阶段暂时不上 `vue-router`？

## 简短结论

因为 Pony Agent 第一阶段需要的是“单页工作台 + 全局状态管理”，而不是多页面网站。`Vue 3 + Pinia` 正好适合承接组件边界和运行时状态，`vue-router` 现在还不是刚需。

## 系统化梳理

### 1. 为什么不是继续原生 DOM

一开始用原生 DOM 搭骨架是够的，但一旦工作台开始有：

- 聊天区
- 运行时状态
- Graph 轨迹
- 工具调用日志
- Provider 状态

状态和视图会迅速复杂，原生 DOM 很快不适合长期维护。

### 2. 为什么是 Vue 3

Vue 3 的优势在于：

- 学习成本低
- 组件边界清晰
- 适合快速搭建结构化工作台
- 和 Tauri 配合自然

### 3. 为什么要 Pinia

Pinia 适合承接这类跨组件的全局状态：

- 当前会话
- 当前运行阶段
- 工具轨迹
- 健康状态
- 当前 turn 结果

如果没有集中状态管理，后面会出现大量跨组件传值和状态碎片。

### 4. 为什么先不上 vue-router

因为第一阶段更像一个桌面工作台，而不是多页面应用。

现在最重要的是：

- 让前端成为 runtime 调试台
- 承接 Rust 状态
- 为后续 turn 链路留出清晰结构

这时候 router 会增加页面结构复杂度，但对当前核心目标帮助不大。

## 常见误区

- 误区 1：只要是 Vue 项目就应该第一时间上 router
- 误区 2：状态少的时候不用 Pinia，后面再说
