# 为什么 session 完善与模块化重构，最终是在落实一套领域级设计哲学
## 问题

我们这轮持续做 `session` 完善、`runtime` 收口，以及 `planner / context / telemetry / turn_flow` 的模块化拆分，本质上体现了什么工程设计哲学？为什么这些哲学不是 Rust 特有技巧，而是跨语言、跨框架都成立的系统设计原则？

## 简短结论

这轮重构背后最核心的，不是“把文件拆开”，而是把系统从“围绕一条流程堆代码”，转成“围绕领域对象、变化边界和长期演进能力组织代码”。

可以压缩成三句话：

- 先识别系统里长期存在的对象，再组织一次次流程
- 按变化原因解耦，而不是按表面功能聚类
- 让核心稳定、状态显式、边界清晰、可观测性内建

所以，我们做的不是语法层面的重构，而是在把 Pony Agent 往真正可演进的 agent core 推进。

## 系统化梳理
### 1. 按领域对象组织，而不是按流程堆叠

流程视角最容易让人先想到：

- 接收输入
- 组织上下文
- 调模型
- 判断是否调工具
- 返回结果
- 持久化历史

这当然是一条真实流程，但它不是系统最稳定的骨架。

真正更稳定的，是系统里长期存在的对象：

- `session`
- `turn`
- `provider`
- `tool call / tool result`
- `context`
- `trace / telemetry`

流程会不断变化，比如：

- 这一轮是否走 tool
- 这一轮是否 stream
- 这一轮是否 fallback
- 这一轮是否需要恢复旧 session

但这些对象不会因为流程调整而消失。  
所以成熟系统会先回答“系统由什么组成”，再回答“这些对象这一次如何互动”。

这是一条非常通用的工程原则：

- 流程描述“一次发生了什么”
- 对象描述“系统长期由什么组成”

短期看流程更直观，长期看对象更稳定。

### 2. 为什么 `session` 是领域对象，而 `run_turn` 只是流程入口

`session` 是领域对象，因为它满足几个条件：

- 它在时间上持续存在
- 它有自己的状态
- 它会被很多流程重复使用
- 即使外层交互形式变化，它依然存在

在 Pony Agent 里，`session` 承担的是：

- 会话 id
- 历史消息
- summary
- 更新时间
- 会话恢复、切换、删除和持久化的承载点

而 `run_turn()` 更像一次动作：

- 用户发来这一轮输入
- runtime 执行一次回合
- 这一轮结束后，动作就结束了

所以：

- `session` 是长期容器
- `run_turn()` 是容器里的单次事件

如果把 `run_turn()` 当系统中心，就容易把所有复杂性都塞进一个大函数里。  
如果把 `session` 当系统中心，`run_turn()` 就会自然退回成“围绕某个 session 执行本轮”的入口。

这也是为什么我们先做 `SessionStore`，再做对话历史 UI：  
因为会话不是前端装饰，而是 agent core 的真实对象。

### 3. 按变化原因解耦，而不是按表面功能解耦

模块化最容易犯的一个误区，是只看“这些代码现在是不是都在做同一个功能”。

但好的解耦标准其实是：

- 它们以后为什么会被改
- 哪些代码会经常一起改
- 哪些代码只是碰巧出现在同一条流程里

在当前重构里：

- `session` 会因为持久化策略、多会话、恢复机制而变
- `context` 会因为提示词拼接、上下文裁剪、缓存命中策略而变
- `planner` 会因为本地规则和模型决策协同方式而变
- `telemetry` 会因为状态展示、trace 粒度、诊断需求而变
- `turn_flow` 会因为 stream 事件壳、失败事件、turn 完成事件而变
- `runtime` 本身则应主要因为“流程编排方式变化”而变化

这些模块都参与同一个 turn，但它们不是因为同一种原因被修改。  
所以不能因为“都和 run turn 有关”就长期绑在一起。

这条原则可以压缩成一句话：

- 不要只看它们现在像不像一类
- 要看它们未来会不会因为同一个原因一起变化

如果不会，那就应该分开。

### 4. orchestration 和 capability 必须分层

这轮重构的另一个重点，是把 `runtime` 往 orchestration layer 推。

也就是说：

- `runtime` 负责协调顺序和阶段推进
- `planner` 负责回合前的工具/决策策略
- `context builder` 负责组装模型请求
- `tool executor` 负责真正执行工具
- `session store` 负责会话读写和恢复
- `telemetry builder` 负责 trace/tool activity 结构

这体现的是一条跨系统的设计哲学：

不要让“编排层”同时兼任“具体能力实现层”。

否则会出现一个典型问题：

- 想改 stream，要进 runtime
- 想改 tool 策略，要进 runtime
- 想改 trace，要进 runtime
- 想改 session 持久化，还要进 runtime

最后 runtime 就会变成“所有变化的汇流点”。  
这正是我们这轮一直在避免的情况。

### 5. 让状态显式化，而不是让状态隐含在流程里

session 重构还体现了一条重要哲学：

系统不能只围绕当前请求工作，还要显式管理跨请求延续的状态。

对于 agent 系统尤其如此：

- 用户不是只问一轮
- 工具调用结果会影响下一轮
- 会话恢复和切换是真实需求
- summary、history、context window 不是同一层东西

如果这些状态不被单独建模，就会隐含在：

- 前端临时 history
- 某个大函数里的局部变量
- 某个页面状态
- 某次请求的拼装过程

这会让系统越来越难维护。

所以我们现在做的，是把状态从“隐式副产物”提升成“显式一等公民”。  
这也是 `session store`、`session snapshot`、`append_turn()` 这些设计的根本意义。

### 6. 边界先于实现

你一直强调“Rust agent core 是后端、单独可部署的引擎”，这其实也是这轮重构的核心哲学之一。

它意味着：

- 核心能力先在 core 里成立
- Tauri 只是当前桌面适配层
- 前端只负责交互和显示
- 未来接 HTTP / SSE / Web / TUI 时，应该复用同一套 core

这就是“边界先于实现”：

- 先决定哪些东西属于 core
- 再决定当前桌面壳如何接入

这种设计的价值在于，外层会变，但内核应该尽量稳定。  
真正要可复用、可迁移、可跨平台的，不是某个 UI 页面，而是 provider / tool / session / runtime 这些核心对象及其协作方式。

### 7. 可观测性不是补丁，而是系统能力

这轮我们把 `fallback reason`、`trace`、`phase`、`tool activities` 逐步做成结构化数据，也体现了一个成熟系统观：

系统不只是要“能跑”，还要“能解释自己怎么跑的”。

所以：

- 为什么走了这个 provider
- 为什么触发 fallback
- 当前在 turn 的哪个 phase
- 有没有调用 tool
- 首 token 延迟是多少
- token usage 怎样

这些都不该只是临时 debug 输出，而应该成为系统稳定对外暴露的一部分。

这就是为什么 `telemetry` 值得独立出来。  
因为可观测性不是日志残渣，而是产品、诊断和后续运维都会持续依赖的能力层。

## 常见误区

- 误区 1：模块拆得越多，架构就越好。真正重要的不是文件数，而是边界是否对应真实职责和真实变化轴。
- 误区 2：只要一段代码在同一条流程里出现，就应该留在同一个文件里。很多复杂性恰恰来自“同场出现但变化原因不同”。
- 误区 3：session 只是聊天历史的持久化壳。实际上 session 是 agent 长期语境、恢复能力和跨轮状态的承载对象。
- 误区 4：可观测性是后面补的东西。对 agent 系统来说，缺少可观测性，很多 provider/tool 问题都无法稳定诊断。

## 后续值得继续学什么
- `turn` 和 `session` 的关系是什么，为什么它们不是同一层对象
- `runtime` 作为 orchestration layer，未来应该继续保持多薄才合适
- `context builder`、`session summary` 和 prompt caching 未来如何协同
- stream / non-stream 两条路径未来在哪一层收敛最合理

## 可延展内容选题

- 为什么 agent 系统应该先围绕领域对象建模，而不是先写一条大流程
- 如何判断一次重构是在做真实架构优化，还是只是在做文件切分
- 从 session、tool、provider 到 runtime：一套可迁移的 agent core 设计哲学
