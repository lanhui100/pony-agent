# Pony Agent 上下文压缩设计原则

## 文档目的

这份文档用于约束 Pony Agent 未来的上下文压缩、摘要、历史裁剪和消息重写策略。

核心目标不是单纯缩短上下文，而是在以下目标之间取得平衡：

- 保留足够任务信息
- 保持前缀稳定
- 提高 cache 命中率
- 降低延迟与成本

## 总原则

### 1. 压缩不是越早越好

在上下文仍然健康时，优先保留原始消息结构，不要过早摘要。

原因：

- 原始历史往往更利于保持稳定前缀
- 过早压缩容易让前缀在每轮都变化
- 会降低 cache 复用价值

### 2. 优先追求“稳定压缩”，而不是“频繁压缩”

一旦某段历史已经压成摘要，应尽量保持它稳定，不要下一轮再把同一段内容重新改写。

原因：

- 摘要一旦反复变化，token 前缀也会反复变化
- 这会直接破坏缓存命中

### 3. 压缩应尽量在明确边界发生

压缩应发生在稳定区段，而不是每一轮都重写整个历史。

推荐边界：

- 超过上下文阈值时
- 某段任务已经完成时
- 某一批工具调用已经闭环时
- 某个阶段性讨论已经结束时

### 4. 保持前缀分层

上下文应尽量拆成不同稳定度的层，而不是视为一个整体。

建议分层：

- 稳定层：system prompt、长期规则、固定摘要
- 半稳定层：最近若干轮关键对话
- 易变层：本轮工具结果、临时附件、即时提示

压缩时应优先只动某一层，避免整段重写。

### 5. 不要为了变短而损失可复用前缀

每次压缩设计都要先评估：

- token 变少了多少
- 前缀稳定性损失了多少
- 下一轮是否更难命中 cache

只有“综合收益”为正时，压缩才值得做。

## 具体策略建议

## 策略一：延迟压缩

在上下文长度还没有逼近窗口时，不主动压缩。

适用场景：

- 会话前中期
- 问题还在快速演化
- 最近轮次仍频繁被引用

## 策略二：边界式压缩

把较老、较稳定的一段历史一次性压成摘要，然后尽量固定住。

适用场景：

- 一个子任务已经结束
- 一批工具调用已经完成
- 某个问题已经形成阶段性结论

## 策略三：摘要冻结

已生成的历史摘要，默认不在每轮继续重写。

只有在出现这些情况时才重新生成：

- 原摘要已明显过时
- 当前任务目标发生重大变化
- 压缩层自身也开始膨胀

## 策略四：局部压缩优先于全局重写

优先只压缩：

- 工具结果
- 很长的附件
- 已完成分支
- 冗长但低复用的中间文本

尽量不要直接改写整个会话前缀。

## 策略五：压缩结果需要可观测

压缩不是隐形动作，未来应能在 trace 或 runtime 状态中体现：

- 为什么压缩
- 压了哪一段
- 采用了哪种压缩方式
- 是否生成了新的稳定摘要

这有助于调试，也有助于理解 cache 命中变化。

## 不推荐的做法

- 每轮都重写完整摘要
- 在上下文还不长时就频繁 compact
- 把最近高价值消息过早压成抽象总结
- 不区分稳定层和易变层，整段统一重写
- 只看 token 变短，不看后续 cache 命中损失

## 对 run_turn 的未来要求

未来 `run_turn()` 发展到上下文管理阶段时，压缩逻辑应满足：

- 压缩前能判断是否值得压
- 压缩后能产出稳定摘要或边界
- 不破坏用户主任务的连续性
- 尽量保留后续轮次可复用的 prompt 前缀

## 当前阶段的落地建议

在 Pony Agent 的早期实现阶段，建议采用以下默认策略：

1. 暂不做自动压缩
2. 先把 `run_turn()` 的消息结构、工具结果和会话摘要跑通
3. 等多轮会话与工具调用稳定后，再引入第一版边界式压缩
4. 第一版压缩优先处理工具结果和老历史，不动最近轮次

## 一句话原则

好的上下文压缩，不是“尽量压短”，而是“尽量在不破坏稳定前缀的前提下压短”。
