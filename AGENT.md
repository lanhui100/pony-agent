# Pony Agent

## 项目介绍

Pony Agent 是一个基于 `Tauri + Rust + Vue + TypeScript` 重构的桌面智能体项目。

本项目采用“学习模式重构”：

- 保留 `hermes/` 作为参考实现
- 在根目录构建新的 Pony Agent 主工程
- 逐步迁移智能体核心能力，而不是一次性重写

当前项目目标不是复刻 Hermes 的全部功能，而是先构建一个结构清晰、可观察、可迭代的智能体运行时，并在此基础上逐步补齐：

- 对话主循环
- 工具调用
- 会话状态
- Graph 编排
- 记忆系统
- Provider 抽象

## 当前目录约定

- `hermes/`：Hermes 原始参考实现，只读参考区
- `src/`：前端界面层
- `src-tauri/`：Tauri 容器与 Rust 智能体核心
- `docs/`：项目文档体系

## 技术栈

### 当前已落地

- `Tauri v2`
- `Rust`
- `shadcn-vue` 风格组件
- `Tailwind CSS`
- `Vite`
- `TypeScript`

### 目标前端栈

- `Vue 3`
- `TypeScript`
- `Vite`
- `Pinia`
- `Tailwind CSS`
- `shadcn-vue`

### 暂不优先

- `vue-router`
- TUI 作为主交互层

说明：

- 第一阶段使用 Tauri UI 作为智能体运行时调试台
- 如果后续需要开发者模式或远程终端场景，再考虑补充 TUI

## 架构原则

1. 参考实现与新实现分离
2. 先打通最小闭环，再扩展复杂能力
3. 智能体核心优先，界面体验第二优先
4. 可观察性优先于炫技
5. 统一抽象优先于临时堆叠

## 最佳实践

### 通用

- 优先小步提交、小步验证
- 每次只引入一个主要抽象
- 每个模块都必须能回答“它负责什么，不负责什么”
- 文档要和代码一起演进

### Rust

- 先定义领域模型和 trait，再接具体实现
- 避免一开始把 provider、tool、session 强耦合
- 优先显式状态机和清晰数据结构
- 错误类型要可追踪、可展示

### 前端

- UI 先服务于调试与观察，不急于做复杂产品包装
- 全局状态进入 Pinia，本地展示状态留在组件内部
- 前后端通信统一通过 Tauri command / event 边界

### 文档

- 项目事实写入 `AGENT.md`
- 长文档进入 `docs/`
- 重大架构决策要记录“为什么”
- 复杂开发任务默认先建立 OpenSpec 变更，再进入实现
- 任务计划、进度、续作和归档不写入 `AGENT.md`
- 学习问题、系统化梳理和内容素材沉淀写入 `docs/learning/`
- 文档文件名尽量稳定，不频繁改名

## 规范约束

### 代码组织

- 不直接在 `hermes/` 中开发 Pony Agent 新功能
- 新功能默认落在根目录主工程
- 参考 Hermes 时，优先提炼思想，不直接复制复杂历史实现

### 命名

- 项目正式名：`Pony Agent`
- 包名、标识符、窗口标题统一使用 `Pony Agent` 及其对应 slug

### 实施顺序

- 先前端框架与调试台
- 再 Rust 单轮运行时
- 再工具系统
- 再会话与记忆
- 最后扩展多代理和复杂编排

### 决策原则

遇到分歧时，优先级如下：

1. 是否更利于理解系统
2. 是否更利于演进
3. 是否更利于调试
4. 是否更利于最终产品体验
